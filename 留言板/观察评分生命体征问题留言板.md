# 观察、评分与生命体征显示问题留言板

更新日期：2026-06-19  
用途：记录“点击第一眼观察后对话记录消失、提交分诊后没有评分、已观察多项显示未描述、生命体征血糖以下疑似乱码”的修复方法和实施步骤，供后续 AI 或开发者接手。

## 一、问题范围

本留言板覆盖以下用户反馈：

1. 在训练页点击“第一眼观察/记录观察”后，对话记录全部消失。
2. 提交分诊后没有看到评分。
3. “已观察”里除“一般状态”外，呼吸状态、皮肤灌注、意识状态等经常显示“未描述”，也不清楚应在哪里补描述才会被记录。
4. 生命体征中血糖以下的部分项目疑似乱码或显示异常。

相关主要文件：

- `frontend/src/pages/triage/TriageTraining.jsx`

- `frontend/src/components/Toast.jsx`
- `frontend/src/components/useToast.js`

## 二、当前链路简图

训练页初始化：

1. `TriageTraining.jsx` 读取 URL 中的 `case`、`mode`、`task_id`、`time_limit`。
2. 调用 `getTriageCaseDetail(caseId)` 获取安全病例。
3. 调用 `startTriageTraining(caseId, ...)` 创建训练记录。
4. 前端用返回的 `record_id` 和 `opening_line` 初始化本地 `recordId`、`messages`。

第一眼观察：

1. 前端 `handleObserve()` 调用 `observeTriagePatient(recordId, ids)`。
2. API 实际请求 `POST /api/triage/training/{record_id}/observe`。
3. 后端 `observe_patient()` 从病例中拼出 `appearance`、`breathing_effort`、`skin_perfusion`、`consciousness`。
4. 后端当前只把观察 id 写入 `record.actions` 和 `record.observed_items`。
5. 前端只用接口返回的 `observations` 更新本地 `observed`。

生命体征：

1. 前端 `handleMeasure()` 调用 `measureTriageVitals(recordId, ids)`。
2. 后端 `measure_vitals()` 调用 `get_vital_signs(case, ids)`。
3. `get_vital_signs()` 直接返回病例 JSON 中的 `required_measurements` 对象。
4. 前端直接显示 `v.label` 和 `v.value`，当前没有统一格式化 value，也没有显示 unit。

提交评分：

1. 前端 `handleSubmit()` 调用 `submitTriage(recordId, { level, zone, disposition })`。
2. 后端 `submit_triage()` 先写入分诊选择，再调用评分器。
3. 后端 `save_score()` 应把 `total_score`、`score_detail`、`feedback` 等写回训练记录。
4. 前端跳转 `/triage/record/{recordId}`，并通过 `location.state` 带上 `score` 与 `record`。
5. 报告页 `TriageRecordDetail.jsx` 再调用 `getTriageRecord(id)` 重新拉取记录并恢复评分。

## 三、问题 1：点击第一眼观察后对话记录消失

### 初步判断

最可疑根因在 `TriageTraining.jsx` 的初始化 `useEffect`：

```jsx
useEffect(() => {
  ...
  getTriageCaseDetail(caseId)...
  startTriageTraining(caseId, ...)...
}, [caseId, mode, taskIdParam, timeLimitParam, toast, navigate]);
```

这里把 `toast` 对象放进依赖数组。`useToast()` 当前返回的是 `ToastContext` 的整个对象，而 `ToastProvider` 每次 toast 状态变化都会重新渲染并创建新的 context value 对象。点击“记录观察”后会调用 `toast.success("观察已记录")`，这会触发 ToastProvider 状态变化，进而让 `toast` 对象引用变化，训练页初始化 effect 可能重新执行。

一旦初始化 effect 重新执行，就会再次调用 `startTriageTraining()`，生成新的训练记录，并执行：

```jsx
setRecordId(data.record_id);
setMessages([{ role: "patient", content: data.opening_line }]);
```

表现上就是：原来的对话记录被重置，只剩新开场白，看起来像“对话记录全消失”。

开发环境如果启用了 React StrictMode，初始化 effect 还可能被额外触发两次，进一步放大这个问题。

### 修复步骤

1. 修改 `frontend/src/components/Toast.jsx`，给 Provider 的 value 加 `useMemo`，保证 context value 在方法引用不变时稳定：

```jsx
import { useMemo } from "react";

const value = useMemo(
  () => ({ toast, success, error, warning, info }),
  [toast, success, error, warning, info],
);

return (
  <ToastContext.Provider value={value}>
    {children}
    ...
  </ToastContext.Provider>
);
```

2. 修改 `frontend/src/pages/triage/TriageTraining.jsx`，不要把整个 `toast` 对象作为初始化 effect 的依赖。建议拆出稳定方法：

```jsx
const { success, error, warning } = useToast();
```

对应替换：

```jsx
toast.success(...) -> success(...)
toast.error(...) -> error(...)
toast.warning(...) -> warning(...)
```

初始化 effect 依赖改为：

```jsx
}, [caseId, mode, taskIdParam, timeLimitParam, error, navigate]);
```

3. 增加一次性启动保护，避免训练记录被重复创建：

```jsx
const startedRef = useRef(false);

useEffect(() => {
  if (!caseId) { navigate("/triage"); return; }
  if (startedRef.current) return;
  startedRef.current = true;
  ...
}, [caseId, mode, taskIdParam, timeLimitParam, error, navigate]);
```

4. 更稳妥的做法是拆分两个 effect：

- 一个 effect 只负责加载病例详情。
- 一个 effect 只负责创建训练记录。

这样观察、测量、toast、右侧面板状态变化都不应该重新触发“创建训练”。

5. 在 `handleObserve()` 成功后只更新 `observed`，不要触碰 `messages`、`recordId`、`caseData`：

```jsx
const { data } = await observeTriagePatient(recordId, ids);
setObserved((prev) => mergeObservation(prev, data.observations || []));
success("观察已记录");
```

### 验证方法

1. 打开训练页，发送 2 轮对话。
2. 选择“一般状态”和“呼吸状态”，点击“记录观察”。
3. 期望左侧对话仍保留所有历史消息，不生成新开场白。
4. 浏览器 Network 面板中，点击观察后只应出现一次 `/observe` 请求，不应再次出现 `/training/start` 请求。
5. 后端 `backend/triage_data/records` 中不应因一次观察额外生成新 record。

建议增加前端回归测试：

- mock `startTriageTraining`、`observeTriagePatient`。
- 渲染 `TriageTraining` 后确认 start 只调用一次。
- 点击观察并触发 toast 后，确认 start 仍只调用一次，messages 仍包含观察前的对话。

## 四、问题 2：提交分诊后没有评分

### 后端当前状态

后端提交评分链路基本存在：

- `backend/routers/triage.py` 的 `submit_triage()` 会调用评分器。
- 有 `scoring_rubric` 时调用 `score_triage_real_rubric()`。
- 动态时间线病例调用 `score_triage_v4()`。
- 有 `dialogue_state_machine` 时调用 `score_triage_v3()`。
- 最后调用 `save_score(record_id, score_result)` 写回记录。
- 接口返回 `{"record": record, "score": score_result}`。

所以“没有评分”优先排查前端展示与记录恢复，而不是先怀疑评分器不存在。

### 前端可疑点

`TriageRecordDetail.jsx` 中恢复评分的判断是：

```jsx
if (rec?.total_score !== null && rec?.score_detail) {
  setScore(...)
}
```

这个判断不够严谨：

- `undefined !== null` 为 true。
- 如果接口返回字段不完整，会出现错误状态判断。
- 如果导航 state 里有 `score`，随后 `getTriageRecord()` 返回的记录暂时没有 `score_detail`，可能导致页面表现不稳定。

另外，考核模式下：

```jsx
const canShowFeedback = !isExamMode || record?.show_feedback_immediately !== false;
```

如果任务配置不允许立即展示反馈，页面会隐藏详细评分。用户可能把“反馈被隐藏”理解为“没有评分”。

### 修复步骤

1. 后端确认提交接口返回完整 score：

在 `backend/routers/triage.py` 的 `submit_triage()` 评分后增加防御性校验：

```python
if not isinstance(score_result, dict) or score_result.get("total_score") is None:
    raise HTTPException(status_code=500, detail="评分结果生成失败")
```

2. 确保 `save_score()` 后重新加载的是已评分记录：

```python
save_score(record_id, score_result)
record = get_record(record_id, user_id=current_user.id)
if record.get("total_score") is None:
    raise HTTPException(status_code=500, detail="评分保存失败")
```

3. 修改 `TriageRecordDetail.jsx` 的评分恢复判断：

```jsx
if (rec?.total_score != null && rec?.score_detail) {
  setScore(...)
}
```

4. 不要让一次失败的重新拉取覆盖导航带来的 score。建议：

```jsx
if (navScore) setScore(navScore);
...
if (rec?.total_score != null && rec?.score_detail) {
  setScore(...)
} else if (!navScore) {
  setScore(null);
}
```

5. 考核模式下即使隐藏详细反馈，也应显示“评分已生成/待教师发布”的明确状态。可以保留总分或显示：

```jsx
本次为考核模式，系统已完成评分，详细反馈将在教师发布后开放。
```

6. 在 `handleSubmit()` 中，如果接口返回没有 `data.score.total_score`，直接 toast 错误并不要跳转：

```jsx
if (data?.score?.total_score == null) {
  error("提交成功但评分结果为空，请联系教师或重试");
  return;
}
```

### 验证方法

后端：

```powershell
cd backend
pytest -q
```

新增接口测试：

1. 创建训练记录。
2. 测量至少一个生命体征。
3. 提交分诊等级和区域。
4. 断言响应中 `score.total_score` 不是空。
5. 断言重新读取 record 时 `record.total_score`、`record.score_detail` 均存在，`record.status == "scored"`。

前端：

1. 训练页提交后应跳转报告页。
2. 报告页应显示总分。
3. 刷新报告页后仍能从 `/training/records/{record_id}` 恢复总分。
4. 考核模式如果不显示详细反馈，也必须出现“评分已生成，反馈暂不开放”的文字。

## 五、问题 3：已观察中多项显示“未描述”，以及描述应该写在哪里

### 当前根因

后端 `observe_patient()` 当前硬编码了四项观察：

```python
OBS_MAP = {
    "appearance": {"value": pp.get("appearance", "未描述")},
    "breathing_effort": {"value": f"呼吸{vs.get('respiratory_rate_bpm','')}次/分" if vs.get('respiratory_rate_bpm') else "未描述"},
    "skin_perfusion": {"value": vs.get("skin_perfusion", "未描述")},
    "consciousness": {"value": vs.get("consciousness", "未描述")},
}
```

而病例数据里更常见的结构是：

- `patient_profile.appearance` 有“一般状态”。
- `required_measurements` 里有 `respiratory_rate_bpm`、`consciousness`、`skin_perfusion` 等测量项。
- 部分病例未把这些值同步放到顶层 `vital_signs`。

因此后端只从 `vital_signs.skin_perfusion`、`vital_signs.consciousness` 取值时，就会显示“未描述”。

同时，当前 `record_action()` 只保存：

```python
{"observation_ids": [o["id"] for o in observations]}
```

训练记录里没有保存观察当时的 `label/value/is_abnormal/source`，后续回放或评分证据也无法还原完整观察内容。

### 描述应该写在哪里

建议统一病例数据规范：

1. 一般状态：

写在：

```json
"patient_profile": {
  "appearance": "神志清楚，无痛苦面容，无气促。"
}
```

2. 呼吸状态：

优先写在 `required_measurements` 中：

```json
{
  "id": "respiratory_rate_bpm",
  "label": "呼吸",
  "value": "18",
  "unit": "次/分",
  "source_field": "vital_signs.respiratory_rate_bpm"
}
```

如有呼吸费力、端坐位、辅助呼吸肌使用等非数值描述，可新增：

```json
{
  "id": "breathing_effort",
  "label": "呼吸状态",
  "value": "轻度气促，可完整说话",
  "unit": "",
  "source_field": "vital_signs.breathing_effort"
}
```

3. 皮肤灌注：

写在 `required_measurements`：

```json
{
  "id": "skin_perfusion",
  "label": "皮肤灌注",
  "value": "皮肤温暖，末梢灌注可，无发绀",
  "unit": "",
  "source_field": "vital_signs.skin_perfusion"
}
```

4. 意识状态：

写在 `required_measurements`：

```json
{
  "id": "consciousness",
  "label": "意识状态",
  "value": "清醒，可正确应答",
  "unit": "",
  "source_field": "vital_signs.consciousness"
}
```

也可以同时保留 GCS：

```json
{
  "id": "gcs_score",
  "label": "GCS",
  "value": "15",
  "unit": "分",
  "source_field": "vital_signs.gcs_score"
}
```

### 修复步骤

1. 在 `backend/routers/triage.py` 中新增通用取值函数，优先从 `required_measurements` 按 id 取，再从 `vital_signs` 取：

```python
def _measurement_by_id(case, mid):
    for item in case.get("required_measurements", []):
        if item.get("id") == mid:
            return item
    return None

def _display_measurement(item):
    if not item:
        return ""
    value = item.get("value")
    unit = item.get("unit") or ""
    if value is None or value == "":
        return ""
    return f"{value}{unit}" if unit and str(value).strip() else str(value)
```

2. 改造 `OBS_MAP`：

```python
rr = _measurement_by_id(case, "respiratory_rate_bpm")
skin = _measurement_by_id(case, "skin_perfusion")
cons = _measurement_by_id(case, "consciousness") or _measurement_by_id(case, "gcs_score")

OBS_MAP = {
    "appearance": {
        "label": "一般状态",
        "value": pp.get("appearance", "未描述"),
        "is_abnormal": False,
        "source": "patient_profile.appearance",
    },
    "breathing_effort": {
        "label": "呼吸状态",
        "value": _display_measurement(rr) or pp.get("appearance", "未描述"),
        "is_abnormal": bool(rr and rr.get("is_abnormal")),
        "source": "required_measurements.respiratory_rate_bpm",
    },
    "skin_perfusion": {
        "label": "皮肤灌注",
        "value": _display_measurement(skin) or "未描述",
        "is_abnormal": bool(skin and skin.get("is_abnormal")),
        "source": "required_measurements.skin_perfusion",
    },
    "consciousness": {
        "label": "意识状态",
        "value": _display_measurement(cons) or "未描述",
        "is_abnormal": bool(cons and cons.get("is_abnormal")),
        "source": "required_measurements.consciousness",
    },
}
```

3. 保存完整观察详情，而不只是保存 id：

```python
record_action(record_id, "observe_patient", {
    "observation_ids": [o["id"] for o in observations],
    "observations": observations,
})
record["observed_items"] = list(existing)
record["observed_details"] = observations
```

4. `get_training_state()` 建议返回：

```python
"observed_items": record.get("observed_items", []),
"observed_details": record.get("observed_details", []),
```

5. 前端 `TriageTraining.jsx` 成功观察后不要覆盖旧观察，应该合并：

```jsx
setObserved((prev) => {
  const map = new Map(prev.map((item) => [item.id, item]));
  (data.observations || []).forEach((item) => map.set(item.id, item));
  return Array.from(map.values());
});
```

6. 前端显示“未描述”时，应同时提示“病例数据未配置该观察项”，避免用户误以为自己操作错了。

## 六、问题 4：生命体征血糖以下部分显示乱码或异常

### 当前观察

部分病例在 `required_measurements` 的血糖后面还有 `other_assessments` 等项目，其 value 不是普通展示文本，而是 Python 列表字符串，例如：

```json
"value": "['孕周', '末次月经', '阴道流血量和颜色', '腹痛部位', '肩痛', '晕厥', '既往宫外孕或流产史', '妇科手术史', '已确认宫内妊娠', '生命体征趋势和隐私保护']"
```

以及：

```json
"value": "['饮酒量', '混合药物或其他物质', '呕吐次数', '气道保护', '误吸风险', '头部外伤', '瞳 孔', '低血糖', '低氧', '自伤', '被他人伤害', '同行人员身份和可靠性', '1.']"
```

这类内容在页面里直接渲染，会像“乱码”或“不像正常生命体征”。这不一定是 UTF-8 编码损坏，更像是数据格式错误或未做显示格式化。

另外前端目前只显示：

```jsx
{v.value}
```

没有显示 `v.unit`，也没有处理数组、对象、Python repr 字符串、空值。

### 修复步骤

1. 后端先做输出格式化，建议在 `backend/services/triage_patient_v1.py` 的 `get_vital_signs()` 返回前清洗每个 measurement。

新增：

```python
import ast

def _format_measurement_value(value):
    if value is None:
        return ""
    if isinstance(value, list):
        return "、".join(str(x).strip() for x in value if str(x).strip())
    if isinstance(value, str):
        text = value.strip()
        if text.startswith("[") and text.endswith("]"):
            try:
                parsed = ast.literal_eval(text)
                if isinstance(parsed, list):
                    return "、".join(str(x).strip() for x in parsed if str(x).strip())
            except Exception:
                return text
        return text
    return str(value)

def _normalize_measurement(m):
    item = dict(m)
    item["value"] = _format_measurement_value(item.get("value"))
    item["display_value"] = item["value"]
    unit = item.get("unit") or ""
    if unit and item["display_value"]:
        item["display_value"] = f"{item['display_value']} {unit}"
    return item
```

然后 `get_vital_signs()` 返回 `_normalize_measurement(m)` 后的结果。

2. 前端 `TriageTraining.jsx` 显示生命体征时优先用 `display_value`，否则拼接 `value + unit`：

```jsx
const valueText = v.display_value || `${v.value ?? ""}${v.unit ? ` ${v.unit}` : ""}`;
```

3. 对 `other_assessments` 这类不是生命体征数值的项目，建议不要放在“生命体征”列表底部。可二选一：

- 移到“其他评估”分组。
- 在 `measurement_options` 中保留，但前端按 `category` 分组显示。

4. 数据层清理：

- 将 Python 列表字符串改为 JSON 数组，或改成普通中文句子。
- 去掉词内异常空格，例如 `瞳 孔` 改为 `瞳孔`。
- 删除无意义尾项，例如 `"1."`。

推荐格式：

```json
{
  "id": "other_assessments",
  "label": "其他评估",
  "value": ["饮酒量", "混合药物或其他物质", "呕吐次数", "气道保护", "误吸风险", "头部外伤", "瞳孔"],
  "unit": "",
  "category": "other_assessment"
}
```

或者：

```json
{
  "id": "other_assessments",
  "label": "其他评估",
  "value": "需评估饮酒量、混合药物、呕吐误吸、头部外伤、瞳孔和自伤/被伤害风险",
  "unit": "",
  "category": "other_assessment"
}
```

### 验证方法

1. 选择包含 `other_assessments` 的病例，例如 `TRIAGE-037` 或 `TRIAGE-049`。
2. 测量全部项目。
3. 期望页面不再显示 `['...']` 这种原始列表字符串。
4. 血糖显示为类似 `5.2 mmol/L`。
5. 其他评估显示为中文列表或正常句子。

建议新增测试：

- 后端单元测试：`get_vital_signs()` 输入 Python 列表字符串，输出不含原始 `[`、`]`、单引号。
- 前端测试：生命体征渲染时优先显示 `display_value`，并显示 unit。

## 十、注意事项

1. 不要把 PowerShell 里显示的中文问号或乱码直接判断为源码编码损坏。先用 `Get-Content -Encoding UTF8` 或 Python `Path(...).read_text(encoding="utf-8")` 确认。
2. “未描述”不是前端录入入口缺失造成的，而是后端没有从正确数据源取到观察值，或病例 JSON 确实缺少该观察项。
3. 当前“第一眼观察”是学生动作记录，不是自由文本录入。若产品希望学生自己描述观察内容，需要新增独立字段，例如 `student_observation_note`，不要混入病例标准观察值。
4. 生命体征和其他评估最好分组显示，不要把所有 `required_measurements` 都塞进“生命体征”一个列表。
5. 评分是否展示要区分“评分未生成”和“考核模式隐藏反馈”。这两个状态必须在 UI 上明确区分。
