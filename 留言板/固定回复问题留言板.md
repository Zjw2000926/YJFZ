# 固定回复问题专项留言板

> **状态：📦 历史问题记录（已归档）— 参见 `观察评分生命体征问题留言板.md` 了解最新修复**

更新日期：2026-06-19  
问题描述：训练对话中，无论护士问什么，虚拟患者经常回答同一句“搬东西后胸口这边刺痛，按着更痛。”  
目标读者：后续接手修复该问题的 AI 或开发者。

## 一句话结论

这不是前端输入框的问题，主要是病例问答数据和后端 slot 匹配策略共同导致的。

以 `TRIAGE-006` 为例，病例数据中多个 `required_questions.answer` 和多个 `dialogue_state_machine.slots[].answer_facts` 都被填成了同一句“搬东西后胸口这边刺痛，按着更痛。”。后端收到问题后按 intent 找 slot，并直接返回第一个匹配 slot 的 `answer_facts[0]`，因此不同问题会命中不同 slot，但这些 slot 的答案内容相同，最终表现为“问什么都回答同一句”。

## 当前复现证据

### 1. 固定回答来源

固定回答来自病例文件：

- `backend/triage_data/cases/TRIAGE-006.json`
- 病例标题：`搬重物后胸壁刺痛`
- `initial_exposure.opening_line` 是 `“搬东西后胸口这边刺痛，按着更痛。”`

同一句还重复出现在：

- `required_questions.q_001.answer`
- `required_questions.q_002.answer`
- `required_questions.q_003.answer`
- `required_questions.q_005.answer`
- `required_questions.q_007.answer`
- `required_questions.chief_complaint.answer`
- `dialogue_state_machine.slots.slot_001.answer_facts`
- `dialogue_state_machine.slots.slot_002.answer_facts`
- `dialogue_state_machine.slots.slot_003.answer_facts`
- `dialogue_state_machine.slots.slot_005.answer_facts`
- `dialogue_state_machine.slots.slot_007.answer_facts`
- `dialogue_state_machine.slots.chief_complaint.answer_facts`

### 2. 直接调用对话服务的复现结果

在关闭 LLM 的规则模式下，用 `TRIAGE-006` 调用 `generate_triage_patient_reply`：

```text
哪里不舒服？ => slot_rule / ask_chief_complaint / slot_007
回复：“搬东西后胸口这边刺痛，按着更痛。”。

什么时候开始疼的？ => slot_rule / ask_onset_time / slot_001
回复：“搬东西后胸口这边刺痛，按着更痛。”。

具体哪个位置疼？ => slot_rule / ask_pain_location / slot_003
回复：“搬东西后胸口这边刺痛，按着更痛。”。

有没有恶心出汗？ => slot_rule / accompanying
回复：没有特别的其他不舒服，就是这个问题。。没有特别的其他不舒服，就是这个问题。。

以前有心脏病吗？ => slot_rule / past_history
回复：平时身体还行，没什么大毛病。。平时身体还行，没什么大毛病。。
```

这说明问题有两层：

1. 一些 slot 的答案被错误地填成同一句主诉。
2. 同一个 slot 可能被同一次提问重复加入，导致一句话重复两遍。

### 3. 不是只有 TRIAGE-006 有问题

快速扫描 50 个病例后发现，全部病例都至少存在以下一种数据质量问题：

- slot answer 与 chief complaint 或 opening line 重复。
- 同一个 intent 映射到多个 slot。
- slot answer 是占位文本 `根据病例资料回答。`

这说明它不是 `TRIAGE-006` 的孤立 bug，而是病例转换或生成流程留下的系统性数据质量问题。

## 根因拆解

## 根因 1：病例数据把多个不同问题的答案写成同一句主诉

文件：`backend/triage_data/cases/TRIAGE-006.json`

示例：

```json
{
  "id": "q_001",
  "label": "胸痛开始时间",
  "answer": "“搬东西后胸口这边刺痛，按着更痛。”"
}
```

`q_001` 应该回答“什么时候开始、持续多久”，而不是重复主诉。合理答案应类似：

```text
刚才搬东西的时候突然开始的，大概十来分钟。
```

再比如：

```json
{
  "slot_id": "slot_003",
  "label": "疼痛性质和具体位置",
  "answer_facts": ["“搬东西后胸口这边刺痛，按着更痛。”"]
}
```

这个 slot 应该回答“疼痛位置 + 性质”，可以改为：

```text
在前胸靠外侧这一块，是刺痛，按压更明显。
```

### 应修复为

每个 slot 必须是“一个问题维度对应一个具体事实”，不要把主诉当成万能答案。

对 `TRIAGE-006` 至少应改成下面这类内容：

```text
chief_complaint: 搬东西后前胸局部刺痛，按着更痛。
onset_time: 刚才搬东西时开始，大概十来分钟。
trauma_detail: 是搬重物后突然出现的，没有摔倒，也没有被撞。
pain_location: 前胸靠外侧这一块，不是胸口正中压榨感。
pain_nature: 刺痛，局部按压更痛。
pain_radiation: 没有放射到肩背、左臂或下颌。
aggravating_relieving: 按压和活动时更痛，安静不动稍好。
accompanying: 没有大汗、恶心、晕厥。
breathing: 没有明显气促，呼吸还可以。
past_history: 没有冠心病、心梗等心血管病史。
medication: 平时没吃特殊药物。
allergy: 没有明确药物过敏。
```

## 根因 2：同一个 intent 映射多个 slot，代码取第一个，容易选错

文件：`backend/triage_data/cases/TRIAGE-006.json`

例如 `ask_onset_time` 同时映射：

```text
slot_001 => “搬东西后胸口这边刺痛，按着更痛。”
onset_time => 大概就是刚才开始的，“搬东西后胸口这边刺痛，按着更痛。”。
```

后端代码在 `backend/services/triage_patient_dialogue.py` 里按 slots 顺序遍历：

```python
for intent in intents:
    for slot in all_slots:
        if intent["intent_id"] in slot.get("canonical_intents", []):
            if slot["slot_id"] not in disclosed:
                matched.append(slot)
                break
```

因此问“什么时候开始”时，会优先命中靠前的 `slot_001`，而不是更语义化的 `onset_time`。

### 应修复为

数据层和代码层都要处理。

数据层：

1. 删除或合并 `slot_001`、`slot_002` 这类由旧 required_questions 转出来的重复 slot。
2. 保留语义稳定的 slot id，例如：
   - `chief_complaint`
   - `onset_time`
   - `pain_location`
   - `pain_nature`
   - `pain_radiation`
   - `severity`
   - `accompanying`
   - `breathing`
   - `past_history`
3. 确保一个 intent 优先只对应一个主 slot。

代码层：

1. 对候选 slot 做评分，而不是拿第一个。
2. 优先选择：
   - 非占位答案。
   - 非 opening_line/chief_complaint 重复答案。
   - slot_id 是语义名称而不是 `slot_001` 这类编号。
   - 没有被 disclosed 的 slot。

建议新增函数：

```python
def _is_placeholder_fact(fact: str) -> bool:
    return not fact or fact.strip() in {"根据病例资料回答。"}

def _is_generic_chief_fact(fact: str, case_data: dict) -> bool:
    initial = case_data.get("initial_exposure", {})
    return fact in {
        initial.get("chief_complaint"),
        initial.get("opening_line"),
    }

def _slot_quality(slot: dict, case_data: dict) -> int:
    facts = slot.get("answer_facts") or []
    first = facts[0] if facts else ""
    score = 0
    if not _is_placeholder_fact(first):
        score += 10
    if not _is_generic_chief_fact(first, case_data):
        score += 10
    if not slot.get("slot_id", "").startswith("slot_"):
        score += 5
    if slot.get("is_required"):
        score += 2
    return score
```

然后匹配时：

```python
candidates = [
    slot for slot in all_slots
    if intent["intent_id"] in slot.get("canonical_intents", [])
    and slot["slot_id"] not in disclosed
]
if candidates:
    matched.append(max(candidates, key=lambda s: _slot_quality(s, case_data)))
```

## 根因 3：同一次提问可能把同一个 slot 加入两次

示例：问“有没有恶心出汗？”

意图可能匹配：

- `ask_accompanying`
- `ask_nausea_vomiting`
- `ask_sweating`

这些 intent 都可能映射到同一个 slot：`accompanying`。

当前代码没有按 `slot_id` 去重，所以会得到：

```text
matched_slots = ["accompanying", "accompanying"]
reply = "没有特别的其他不舒服，就是这个问题。。没有特别的其他不舒服，就是这个问题。。"
```

### 修复步骤

在 `generate_triage_patient_reply` 中，构造 `matched` 时增加 `matched_slot_ids` 集合：

```python
matched = []
matched_slot_ids = set()

...

sid = selected_slot["slot_id"]
if sid not in matched_slot_ids:
    matched.append(selected_slot)
    matched_slot_ids.add(sid)
```

同时 `new_slots` 也要去重：

```python
new_slots = []
for m in matched:
    sid = m["slot_id"]
    if sid not in new_slots:
        new_slots.append(sid)
```

## 根因 4：开启 LLM 不能根治，因为 LLM 只拿到了重复的 allowed info

当前 `.env` 中 `TRIAGE_USE_LLM=true`。这会让系统尝试用 LLM 自然化回复。

但是 LLM prompt 中的“已允许披露的信息”来自 slot 的 `answer_facts`。如果 `answer_facts` 本身就是同一句主诉，LLM 只能围绕这句主诉改写，不能凭空补出正确病史细节。

`backend/triage_data/ai_events.json` 也能看到多条输出都是：

```text
“搬东西后胸口这边刺痛，按着更痛。”
```

### 结论

不要把这个问题理解成“LLM 不够聪明”。先修病例 slot 数据，再让 LLM 做自然化表达。

## 根因 5：旧版 required_questions 没有持续记录 asked_questions

文件：`backend/services/triage_patient_v1.py` 和 `backend/routers/triage.py`

旧版 `match_question` 支持 `asked_questions`，但当前主路由在 V1 fallback 命中后没有把 `question_id` 写回 record。因此旧版 fallback 在某些病例里也可能重复回答同一问题。

### 修复步骤

在 `generate_triage_patient_reply` 返回值中增加 `matched_question_id`，或在 V1 fallback 处把 question id 放进结果：

```python
return {
    ...
    "matched_question_id": v1_result.get("question_id"),
}
```

在 `backend/routers/triage.py` 的 message 接口中保存：

```python
qid = dialog_result.get("matched_question_id")
if qid:
    asked = record.get("asked_questions", [])
    if qid not in asked:
        asked.append(qid)
    record["asked_questions"] = asked
```

不过这只是辅助修复。主要问题仍然是 slot 数据重复和 slot 选择策略。

## 第二步：修后端匹配策略，避免以后被脏数据拖垮

文件：`backend/services/triage_patient_dialogue.py`

需要改动：

1. 增加 slot 质量评分函数。
2. 同一 intent 有多个 slot 时，选择质量最高的 slot。
3. 同一次问题命中的 slot 必须按 `slot_id` 去重。
4. 对 `answer_facts` 为 `根据病例资料回答。` 的 slot，除非没有其他候选，否则不要选择。
5. 组合多个 slot 答案时，去重相同文本，避免重复句子。

建议组合答案逻辑：

```python
facts = []
seen_facts = set()
for m in matched[:2]:
    for fact in m.get("answer_facts", []):
        normalized = fact.strip()
        if normalized and normalized not in seen_facts:
            facts.append(normalized)
            seen_facts.add(normalized)
            break
reply = "。".join(f.rstrip("。") for f in facts) + "。" if facts else UNKNOWN
```

## 第三步：增加病例数据校验脚本

建议新增：


校验内容：

1. 每个 slot 必须有非空 `answer_facts`。

示例校验伪代码：

```python
for case in all_cases:

    chief = initial.get("chief_complaint")
    opening = initial.get("opening_line")
```

## 最终验收标准

修复完成后，至少满足：

1. `TRIAGE-006` 中问“哪里不舒服”“什么时候开始”“哪个位置疼”“有没有恶心出汗”“以前有没有心脏病”，回复应分别落在主诉、起病时间、部位性质、伴随阴性症状、既往史维度。
2. 同一轮回复不得重复同一句两遍。
3. `answer_facts` 不再出现 `根据病例资料回答。`。
4. 除 `chief_complaint` 外，slot 答案不应大量完全等于 opening line。
5. `pytest -q` 通过。
6. 前端真实页面中连续提问 5 个不同问题，患者不再连续回复“搬东西后胸口这边刺痛，按着更痛。”。
