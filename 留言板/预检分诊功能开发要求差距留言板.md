# 预检分诊功能开发要求差距留言板

审查日期：2026-06-19

审查对象：`预检分诊功能开发/预检分诊功能开发.docx`

审查结论：当前系统已经具备预检分诊训练平台的主体骨架，V1 基础闭环、V3 规则引擎、V4 动态病例数据、V5 教师端与 V6 部分平台化能力都有实现痕迹；但它还没有完全达到文档对 V2-V6 的全部验收要求。尤其需要继续强化“基于病例信息的拟人化 LLM 患者”、动态复评前端、专家一致性验证、权限边界、病例编辑审核、导出与并发可靠性。

## 一、已完成或基本可用的部分

1. 基础训练闭环已经形成。系统已有登录、病例选择、问诊、观察生命体征、提交分诊级别与区域、自动评分、训练记录、教师端查看记录等流程。

2. 病例数量已超过 V1/V2/V3 的最低规模。当前病例库合计 50 个病例，其中静态病例 16 个、动态病例 34 个；动态病例数量满足 V4 “至少 15 个动态病例”的要求，但尚未满足 V6 “至少 100 个病例、至少 40 个动态病例”的目标。

3. 评分与规则引擎已经具备基础。后端存在预检分诊规则引擎，能够输出分诊等级、严重错误、规则依据、ESI/CTAS 映射、低估/高估分诊等结果，并已有对应测试。

4. LLM 患者方向已经开始受控化。系统已有受控 LLM 患者相关测试，方向符合文档要求：患者回答应由病例状态机和病例字段约束，而不是由大模型自由编造。

5. 动态病例数据已经具备时间线。动态病例启用数量为 34，且均可统计到至少 2 个时间节点，已经为 V4 等候区观察、病情变化、复评分诊提供数据基础。

6. 教师端与平台化模块已有雏形。系统已有任务、班级/队列、记录、病例审核、场景、分析、导出、学习路径、研究导出、安全审计、AI/语音事件记录等接口或页面。

7. 当前自动化验证状态较好。后端 `pytest -q` 通过 38 个测试；前端 `npm run lint`、`npm run test -- --run`、`npm run build` 均通过。

## 二、与开发文档相比仍未完全达标的问题

### 1. LLM 拟人化还没有达到文档要求的“病例内自然化”

文档要求 V2 的核心不是让 LLM 自由聊天，而是让它在病例状态机、信息披露等级、意图识别和关键问题追踪的约束下，表现为自然患者。当前系统虽然已有受控 LLM 患者的方向，但病例数据层仍存在明显不足：50 个病例的 `patient_variants` 分布均为 1 个，未达到每个病例 2-3 种患者表达变体的要求，也没有充分覆盖家属代答、表达含糊、患者情绪化、信息不完整等模式。

完善方法：

1. 为每个病例新增 `patient_variants`，至少包含标准表达、口语化表达、含糊/焦虑表达三类。

2. 为病例新增或完善 `proxy_mode`、`unclear_expression_mode`、`emotional_state`、`forbidden_disclosure`、`allowed_disclosure_by_stage` 等字段。

3. LLM API 调用必须采用“病例事实白名单 + 状态机 + 意图识别结果 + 信息披露等级”的组合提示词。模型只能改写病例内事实，不能补充病例外诊断、用药、检查结论或现实医疗建议。

4. 增加自动测试：同一问题的不同问法应映射到同一意图；LLM 回答不得泄露未到披露等级的信息；回答不得出现病例 JSON 中不存在的症状、体征、检查、诊断。

5. 增加一套 V2 验收脚本，抽样验证 25 个病例的问答自然度、意图识别准确率、病例事实一致性和关键问题评分点覆盖率。

### 2. V2 的量化验收指标还没有被证明达标

文档要求 V2 达到意图识别准确率不低于 85%、专家评分一致性不低于 85%、每个病例至少 10 个评分点。当前测试能证明部分接口和逻辑可用，但没有看到独立的意图识别评测集、专家一致性评测报告或覆盖全部病例的关键评分点统计。

完善方法：

1. 建立 `triage_eval_intents.jsonl`，每条包含用户问法、标准意图、病例编号、预期披露字段。

2. 建立 `triage_eval_scoring.jsonl`，记录专家标准问诊轨迹、应得分点、严重错误触发条件。

3. 新增后端命令或测试脚本，输出意图准确率、病例事实违规率、评分点召回率、专家一致性。

4. 教师端增加“验收报告”入口，能看到每个版本的评测样本量、通过率、失败样例和修复状态。

### 3. V3 规则引擎仍需补齐阈值和特殊规则

文档要求规则引擎覆盖国内四级预检分诊、ESI/CTAS 映射、生命体征阈值、高危症状、特殊人群升级、低估/高估分诊、严重错误自动识别。当前规则引擎已具备框架，但生命体征阈值主要覆盖血氧、血压、心率、呼吸、体温、疼痛、GCS，文档明确提到的血糖、MEWS 等阈值还未充分体现。特殊人群规则已有部分覆盖，但孕产妇、儿童惊厥、暴力风险、感染暴露、精神/自杀风险等还应继续按文档逐条规则化。

完善方法：

1. 在规则数据中新增血糖、MEWS、休克指数、意识改变、尿量/脱水等阈值。

2. 将文档中的严重错误清单逐项转成机器可执行规则，包括胸痛、卒中、低氧、休克、意识障碍、大出血、过敏反应、重症哮喘、严重创伤、中毒、自杀/暴力、孕产妇危象、儿童惊厥等。

3. 规则输出中保留“命中的规则编号、触发字段、推荐等级、升级原因、映射依据”，方便教师复核和学生复盘。

4. 增加规则回归测试：每条严重错误至少 1 个正例和 1 个反例；低估/高估分诊至少覆盖 I-IV 各等级。

5. 组织专家抽样复核，形成 V3 要求的专家一致性不低于 88%、严重错误识别准确率不低于 95% 的证据。

### 4. V4 动态复评数据可用，但前端操作闭环仍有缺口

文档要求 V4 支持“初次分诊 -> 等候区观察 -> 病情变化 -> 再次评估 -> 升级/维持分诊 -> 时间线报告”。当前动态病例数据足够支撑一部分流程，但动态训练页存在操作风险：测量项状态 `selectedMeasureIds` 没有对应 setter 和可见选择 UI，导致用户在动态页可能无法真正选择并测量生命体征。这样会影响复评、升级分诊和动态评分的实际可用性。

完善方法：

1. 在动态训练页补齐生命体征/观察项选择控件，允许学生选择要测量的项目。

2. 将动态病例的每个时间节点展示为清晰的时间线：初评信息、等待事件、复评生命体征、学生操作、系统判定、升级原因。

3. 提交结果中同时保存初次分诊等级、复评分诊等级、是否及时复评、是否及时升级、复评耗时。

4. 记录页和教师端报告中展示动态时间线，明确学生在哪个节点漏问、漏测、漏升级。

5. 增加前端测试，覆盖动态病例测量、推进时间节点、复评提交、记录回看。

### 5. V5 教师端已成型，但“教学管理平台”要求仍未完全覆盖

文档要求教师端支持学生管理、班级/批次、病例管理、任务发布、考试模式、统计分析、个人能力画像、班级薄弱点、报告导出、教师复评、病例专家审核。当前已有多项雏形，但仍存在不足：病例管理主要偏审核，缺少结构化病例编辑器；报告导出主要是 CSV、HTML/API JSON，未达到 PDF/Excel 导出要求；个人报告和班级报告还不够完整；100 并发学生的验收没有证据；考试模式下答案与提示是否完全隐藏仍需要专项审查。

完善方法：

1. 增加结构化病例编辑器，按文档字段拆分为基本信息、主诉、现病史、生命体征、红旗征象、标准分诊、动态时间线、评分点、严重错误、LLM 披露规则。

2. 病例从“草稿 -> 待专家审核 -> 已通过 -> 可用于训练/考试”流转，未通过专家审核的病例不得进入正式考试库。

3. 增加 PDF 和 Excel 导出，PDF 用于个人/班级报告，Excel 用于教师二次分析。

4. 明确实践、考试、OSCE 三种模式差异：练习可即时反馈，考试隐藏标准答案和提示，OSCE 支持教师复评和过程记录。

5. 增加班级维度仪表盘：高频错误、严重错误分布、低估/高估分诊比例、红旗识别率、关键问题完成率、生命体征完整率、动态复评正确率。

6. 做一次 100 并发学生压测。当前 JSON 文件保存已经使用临时文件替换，能降低写坏文件风险，但缺少文件锁或数据库事务，不能充分证明 100 并发写入下记录不会丢失。

### 6. V6 目前更像平台骨架，尚未达到完整智能训练平台

文档要求 V6 支持多模态虚拟患者、语音、虚拟面容/症状、多人候诊队列、急诊拥挤、情绪与家属不准确信息、暴力/感染风险、特殊人群、个性化学习路径、补救病例、多院区病例库、评分校准、偏差与安全审计、研究导出。当前已有多患者场景、学习路径、研究导出、安全审计、组织/校准/AI 事件/语音事件等基础接口，但很多仍是数据结构或日志层，并非完整训练体验。

当前数量差距：

1. 总病例 50 个，低于 V6 要求的至少 100 个。

2. 动态病例 34 个，低于 V6 要求的至少 40 个。

3. 多患者/复杂场景 8 个，低于 V6 要求的至少 10 个。

4. 场景患者 6 个，仍不足以支撑稳定的多人候诊队列训练。

5. 特殊人群相关关键词可命中较多病例，但这只是关键词统计，不能替代“至少 20 个特殊人群病例”的专家标注验收。

完善方法：

1. 将 V6 拆成阶段交付：先完成多人候诊队列评分，再完成语音问诊，再完成虚拟患者视觉/症状表现，最后做跨院区和研究级导出。

2. 多患者场景需要完整评分：优先级排序、资源分配、漏分高危患者、拥挤状态下复评、绿色通道触发。

3. 语音模块不能只记录事件，应支持语音转文本、意图识别、病例内回答、教师回放、隐私脱敏。

4. 虚拟患者展示应绑定病例字段，例如面色、呼吸困难、出汗、意识状态、疼痛姿势等，不能使用与病例不一致的泛化素材。

5. 建立偏差与安全测试集，覆盖年龄、性别、孕产妇、儿童、老年人、精神危机、暴力风险、感染风险等场景。

6. 所有 AI 回答、评分依据、规则命中、教师改分都要可追踪，满足文档“AI 反馈可追溯”的要求。

### 7. 权限边界仍需加固

部分高敏接口已经限制教师权限，例如训练记录全集、统计、AI 事件读取、组织读取、研究导出、安全审计等；但仍有若干写入或训练类接口需要更严格的角色/归属检查。当前 AI 事件写入、语音事件写入、多人队列开始/提交等接口存在被普通登录用户写入或提交非本人数据的风险，需要补上教师权限或资源归属校验。

完善方法：

1. `POST /ai-events`、`POST /voice-events` 应校验当前用户是否为记录所有者、任务参与者或教师。

2. 多人队列开始/提交应校验场景是否对当前学生开放，提交结果是否属于当前用户。

3. 教师接口统一使用教师权限依赖；学生接口统一使用 owner check。

4. 增加权限测试：学生不能读取他人记录，不能写入他人 AI/语音事件，不能提交未分配任务或未开放场景。

### 8. 旧留言板中有部分结论已经过期，需要归档或更新

现有 `AI问题留言板.md` 中提到的一些问题已经被后续修复或部分修复，例如前后端端口、静态/运行时数据目录拆分、HTML 转义、路由顺序、测试通过情况等。继续保留原文容易让后续开发者误判当前状态。

完善方法：

1. 将旧留言板标注为“历史问题记录”，不要作为最新状态依据。

2. 对已修复项补充修复版本、验证命令和验证结果。

3. 当前新增问题统一收敛到本留言板，后续修复后在本文件追加“修复记录”。

## 三、优先级建议

P0：先修动态训练页测量/复评闭环，否则 V4 的关键用户流程会被前端卡住。

P0：加固 AI/语音事件、多患者队列相关接口的权限和归属校验，避免训练数据被越权写入。

P1：补齐 LLM 患者病例内拟人化能力，包括多表达变体、家属代答、含糊表达、披露等级和防编造测试。

P1：补齐规则引擎阈值和严重错误规则，特别是血糖、MEWS、特殊人群、高危症状和严重错误一票否决。

P1：建立专家一致性与意图识别验收集，把 V2/V3/V4 的百分比指标从“功能存在”推进到“有证据达标”。

P2：完善教师端病例编辑器、PDF/Excel 报告、个人/班级画像、考试/OSCE 模式差异。

P2：为 100 并发学生做压测，并考虑将运行时 JSON 写入迁移到数据库事务或增加文件锁。

P3：按阶段实现 V6 的语音、多模态、多人候诊队列、跨院区病例库、偏差审计与研究导出。

## 四、建议的验收清单

1. 每个病例至少 2-3 个患者表达变体，且 LLM 回答 100% 来自病例白名单字段。

2. 25 个 V2 病例通过自然问答验收，意图识别准确率不低于 85%。

3. 规则引擎可独立判定 I-IV 级，专家一致性不低于 88%，严重错误识别准确率不低于 95%。

4. 动态病例能完整完成初评、等待、复评、升级、记录回看，且报告展示时间线。

5. 教师端能发布任务、管理病例、审核病例、复评学生记录、导出个人和班级报告。

6. 考试模式不显示提示、标准答案和即时解析。

7. 至少完成 100 并发学生压测，证明记录写入、评分、统计不会丢失或串号。

8. V6 达到至少 100 个病例、40 个动态病例、20 个特殊人群病例、10 个多患者场景，并形成专家审核状态。

9. 所有 AI 回答、评分依据、规则命中和教师改分都能追踪到具体病例、用户、时间和版本。
